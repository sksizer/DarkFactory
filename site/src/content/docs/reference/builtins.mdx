Built-in tasks are the atomic operations that DarkFactory provides in the `builtins/` package. They handle git operations, status management, PR review, logging, and cleanup. Workflows compose these builtins alongside agent and shell tasks.

Each builtin is registered in the `BUILTINS` dictionary and dispatched by name from `runner.py`.

## Git Operations

### `ensure_worktree`

Creates a git worktree and branch for the PRD. The branch name is derived from the PRD ID and slug.

No kwargs. Uses `ctx.base_ref` (set from `--base` flag or auto-resolved).

**Behavior**: Creates `.worktrees/<prd-id>-<slug>/` with a new branch `prd/<id>-<slug>` from `ctx.base_ref`. Sets `ctx.worktree_path` and `ctx.cwd`. Acquires a per-PRD advisory file lock at `.worktrees/<prd-id>.lock` for the lifetime of the run. Fails if the worktree already exists but is not safe to resume.

**Dry-run**: Prints the worktree path and branch name that would be created. No filesystem changes.

### `commit`

Stages all changes in the worktree and creates a commit.

| Kwarg | Type | Default | Description |
|-------|------|---------|-------------|
| `message` | `str` | Auto-generated | Commit message. Default: `"feat(<prd-id>): <prd-title>"` |

**Behavior**: Runs `git add -A` in the worktree, then `git commit`. Skips if there are no changes to commit.

**Dry-run**: Prints the diff that would be committed and the commit message. No git operations.

### `push_branch`

Pushes the worktree branch to the remote.

| Kwarg | Type | Default | Description |
|-------|------|---------|-------------|
| `force` | `bool` | `false` | Force-push (used during rework to update existing branches). |

**Behavior**: Runs `git push -u origin <branch>`. Sets upstream tracking.

**Dry-run**: Prints the push command that would execute. No network operations.

### `create_pr`

Creates a GitHub pull request from the worktree branch.

| Kwarg | Type | Default | Description |
|-------|------|---------|-------------|
| `draft` | `bool` | `false` | Create as draft PR. |
| `reviewers` | `str[]` | From PRD | GitHub usernames to request review from. Falls back to PRD `reviewers` field. |
| `base` | `str` | Computed | Target branch for the PR. Usually `main` or the parent PRD's branch. |

**Behavior**: Calls the GitHub API to create a PR. The title is derived from the PRD title, the body from the PRD summary and acceptance criteria. Sets the `pr_url` field on the PRD frontmatter.

**Dry-run**: Prints the PR title, body, base branch, and reviewers. No API calls.

### `cleanup_worktree`

Removes the git worktree after a successful run.

| Kwarg | Type | Default | Description |
|-------|------|---------|-------------|
| `force` | `bool` | `false` | Remove even if the worktree has uncommitted changes. |

**Behavior**: Runs `git worktree remove <path>`. Deletes the local branch if the PR has been created.

**Dry-run**: Prints the worktree path that would be removed. No filesystem changes.

## Lifecycle Management

### `set_status`

Updates the PRD frontmatter status field.

| Kwarg | Type | Required | Description |
|-------|------|----------|-------------|
| `to` | `str` | Yes | Target status: `in-progress`, `review`, `done`, `blocked`, `cancelled`. |

**Behavior**: Rewrites the `status:` and `updated:` lines in the worktree's copy of the PRD file. Updates `ctx.prd.status` in memory so downstream tasks see the new status without re-loading from disk.

**Dry-run**: Prints the proposed transition. No file modifications.

<Aside type="caution">
`set_status` targets the worktree's copy of the PRD, not the source repo. Status changes reach the source repo only via PR merge.
</Aside>

## Quality and Attribution

### `lint_attribution`

Verifies that commits in the worktree have correct attribution.

| Kwarg | Type | Default | Description |
|-------|------|---------|-------------|
| None | | | No configurable parameters. |

**Behavior**: Checks that all commits on the branch have valid author information and are signed (if signing is configured). Fails the workflow if attribution is missing or invalid.

**Dry-run**: Prints attribution status for each commit. No modifications.

## Logging and Artifacts

### `commit_transcript`

Commits the agent transcript to the transcripts directory.

| Kwarg | Type | Default | Description |
|-------|------|---------|-------------|
| None | | | No configurable parameters. |

**Behavior**: Copies the agent's conversation transcript from `.harness-transcripts/<prd-id>.jsonl` to `.darkfactory/transcripts/<prd-id>-<timestamp>.jsonl` in the worktree and commits it.

**Dry-run**: Prints the transcript path. No file operations.

### `commit_events`

Commits the JSONL event log for the current execution.

| Kwarg | Type | Default | Description |
|-------|------|---------|-------------|
| None | | | No configurable parameters. |

**Behavior**: Ensures the event log file for this PRD is flushed and committed to the events directory.

**Dry-run**: Prints the event file path. No file operations.

### `analyze_transcript`

Scans the agent run transcript for problem signals in a two-stage pipeline.

| Kwarg | Type | Default | Description |
|-------|------|---------|-------------|
| None | | | No configurable parameters. |

**Behavior**: Stage 1 runs deterministic Python detectors to produce `Finding` records. Stage 2 fires an LLM narrative (Haiku or Sonnet, config-driven) only when Stage 1 finds issues above a severity threshold. Writes a markdown retro alongside the transcript. Failures in either stage are logged as warnings â€” analysis is advisory and never fatal to the workflow.

**Dry-run**: Prints that transcript analysis would run. No file operations.

### `summarize_agent_run`

Generates a human-readable summary of what the agent did during execution.

| Kwarg | Type | Default | Description |
|-------|------|---------|-------------|
| None | | | No configurable parameters. |

**Behavior**: Analyzes the agent transcript and git diff to produce a summary of changes made, files modified, and tests affected. The summary is included in the PR description.

**Dry-run**: Prints that a summary would be generated. No API calls.

## PR Review

### `fetch_pr_comments`

Loads unresolved PR review threads into the execution context.

| Kwarg | Type | Default | Description |
|-------|------|---------|-------------|
| None | | | No configurable parameters. |

**Behavior**: If `ctx.review_threads` is already populated (pre-fetched by `prd rework`), this is a no-op. Otherwise, calls the `gh` CLI to retrieve unresolved threads from the PR and stores them on `ctx.review_threads`.

**Dry-run**: Prints that PR comments would be fetched. No network operations.

### `reply_pr_comments`

Posts bot replies to addressed PR comment threads.

| Kwarg | Type | Default | Description |
|-------|------|---------|-------------|
| None | | | No configurable parameters. |

**Behavior**: Parses structured reply notes from agent output and posts each as a GitHub comment reply prefixed with `[harness] addressed in <sha>:`. Only executes when `ctx.reply_to_comments` is `True`. Failures are logged as warnings and do not fail the workflow.

**Dry-run**: Prints the replies that would be posted. No network operations.

## Rework

### `rebase_onto_main`

Rebases the worktree branch onto `origin/main`.

| Kwarg | Type | Default | Description |
|-------|------|---------|-------------|
| None | | | No configurable parameters. |

**Behavior**: Fetches `origin/main` then rebases the worktree branch onto it so the agent always works against current mainline code. On conflict, aborts the rebase and fails with the list of conflicting files.

**Dry-run**: Prints the rebase command that would execute. No git operations.

### `fast_forward_branch`

Syncs the local worktree branch with its remote counterpart.

| Kwarg | Type | Default | Description |
|-------|------|---------|-------------|
| None | | | No configurable parameters. |

**Behavior**: Fetches `origin/<branch>` and fast-forwards the local branch to match. Solves the "push rejected" problem when a prior rework cycle or external push has advanced the remote branch.

**Dry-run**: Prints the sync command that would execute. No git operations.

### `resolve_rework_context`

Discovers and populates the worktree path, PR number, rework guard state, and review threads.

| Kwarg | Type | Default | Description |
|-------|------|---------|-------------|
| None | | | No configurable parameters. |

**Behavior**: First task in rework workflows. Populates `ctx.worktree_path`, `ctx.cwd`, `ctx.pr_number`, and `ctx.review_threads`. No-op if the context is already populated by `prd rework`.

**Dry-run**: Prints the context that would be resolved. No filesystem changes.

### `check_rework_guard`

Detects no-change rework loops.

| Kwarg | Type | Default | Description |
|-------|------|---------|-------------|
| None | | | No configurable parameters. |

**Behavior**: Checks whether the current rework cycle produced any file modifications. Records a no-change cycle in the rework guard. Warns below threshold; raises an error at or above threshold to block the commit and push steps.

**Dry-run**: Prints the guard status. No modifications.
