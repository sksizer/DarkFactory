Every PRD is a Markdown file with YAML frontmatter that defines its metadata. The frontmatter drives workflow assignment, dependency resolution, status tracking, and scope computation. This page documents all 20 fields from the PRD dataclass.

## ID Format

PRD IDs must match the regex `^PRD-\d+$`. IDs are natural-sorted: `PRD-9` sorts before `PRD-10`, `PRD-100` before `PRD-1000`.

File naming convention: `PRD-NNN-slug.md` (e.g., `PRD-042-add-retry-logic.md`). The frontmatter `id` field is authoritative -- the filename slug is for human convenience.

## Wikilink Format

Fields that reference other PRDs (`depends_on`, `blocks`, `parent`) use wikilink syntax:

```yaml
depends_on:
  - "[[PRD-100-base-framework]]"
  - "[[PRD-101-data-model]]"
parent: "[[PRD-010-epic-runner-v2]]"
```

The harness resolves wikilinks by matching the numeric ID prefix. The slug portion after the ID is optional but aids readability.

## Complete Field Reference

### Required Fields

| Field | Type | Description |
|-------|------|-------------|
| `id` | `str` | Unique identifier matching `PRD-\d+`. Must be unique across the project. |
| `title` | `str` | Human-readable title. Used in PR titles, commit messages, and CLI output. |
| `status` | `enum` | Current lifecycle status. See Status Transitions below. |
| `kind` | `enum` | PRD type: `epic`, `feature`, `task`, `spike`, `bug`. Determines executability. |

### Optional Fields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `priority` | `enum` | `medium` | Execution priority: `critical`, `high`, `medium`, `low`. Affects ordering within the same DAG depth. |
| `effort` | `enum` | `medium` | Effort estimate: `trivial`, `simple`, `moderate`, `complex`, `xl`. Maps to timeout tiers and model selection. |
| `parent` | `wikilink` | `null` | Parent PRD ID for containment hierarchy. Creates parent-child relationships for epic decomposition. |
| `depends_on` | `wikilink[]` | `[]` | PRD IDs that must be `done` before this PRD is actionable. Drives DAG ordering. |
| `blocks` | `wikilink[]` | `[]` | PRD IDs that this PRD blocks. Inverse of `depends_on` (informational, not enforced). |
| `tags` | `str[]` | `[]` | Freeform tags for filtering, grouping, and workflow matching. |
| `capability` | `str` | `null` | Capability domain (e.g., `auth`, `cli`, `observability`). Used by `capability_multipliers` in config. |
| `workflow` | `str` | `auto` | Explicit workflow override. Bypasses `applies_to` predicate matching. |
| `impacts` | `str[]` | `[]` | Glob patterns for files this PRD will modify. Relative to repo root. Used for conflict detection and scope. |
| `pr_url` | `str` | `null` | GitHub PR URL. Set automatically by the `create_pr` builtin. |
| `branch` | `str` | `null` | Git branch name. Set automatically by `ensure_worktree`. |
| `created` | `date` | `null` | Creation date in ISO 8601 format. |
| `updated` | `date` | `null` | Last modification date. Updated by the harness on status transitions. |
| `assigned_to` | `str` | `null` | GitHub username of the assignee. Informational. |
| `reviewers` | `str[]` | `[]` | GitHub usernames to request review from when creating the PR. |

## Status Transitions

PRDs follow a defined lifecycle state machine:

```
draft → ready → in-progress → review → done
```

Additionally, any status can transition to `blocked` or `cancelled`:

| From | Valid Transitions |
|------|------------------|
| `draft` | `ready`, `blocked`, `cancelled` |
| `ready` | `in-progress`, `blocked`, `cancelled` |
| `in-progress` | `review`, `blocked`, `cancelled` |
| `review` | `done`, `blocked`, `cancelled` |
| `blocked` | `ready` (when blocker is resolved) |

The harness enforces these transitions. The `set_status` builtin rejects invalid transitions (e.g., `draft` directly to `done`).

<Aside type="caution">
Manual edits to the `status` field in a PRD file are not validated until the next `prd plan` or `prd execute` invocation. Use `prd status set PRD-NNN <status>` to ensure transition validity.
</Aside>

## Kind and Executability

| Kind | Executable | Typical Use |
|------|-----------|-------------|
| `epic` | No (unless leaf) | Top-level container. Decomposed by planning workflows into features/tasks. |
| `feature` | No (unless leaf) | Mid-level grouping. Contains tasks. |
| `task` | Yes | Leaf-level work item. The agent implements this. |
| `spike` | Yes | Investigation or research. Agent produces findings. |
| `bug` | Yes | Defect fix. Agent receives reproduction steps and expected behavior. |

The `is_runnable` predicate: `kind in ("task", "spike", "bug") OR is_leaf(prd)`. A `feature` with no children is treated as executable.

## Full Example

```yaml
---
id: PRD-042
title: Add retry logic to task runner
status: ready
kind: task
priority: high
effort: moderate
parent: "[[PRD-010-epic-runner-v2]]"
depends_on:
  - "[[PRD-038-config-schema]]"
  - "[[PRD-039-error-types]]"
blocks:
  - "[[PRD-050-integration-suite]]"
tags:
  - reliability
  - runner
capability: core
impacts:
  - "src/darkfactory/runner.py"
  - "src/darkfactory/config.py"
  - "tests/test_runner_retry.py"
reviewers:
  - alice
  - bob
created: 2026-04-01
---

## Summary
Add configurable retry logic to the task runner...
```

<Aside type="tip">
Use `prd validate PRD-042` to check frontmatter against the schema without executing anything. This catches missing required fields, invalid enum values, and dangling wikilink references.
</Aside>
