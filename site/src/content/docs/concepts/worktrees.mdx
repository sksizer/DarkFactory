title: Worktree Isolation
description: Each PRD executes in its own git worktree, providing filesystem isolation, safe parallel execution, and clean PR boundaries.

Each PRD run creates a **git worktree** -- a separate working directory with its own branch. This provides filesystem isolation between concurrent PRD executions.

## Directory Layout

Worktrees are created at `.worktrees/{id}-{slug}/` with branches named `prd/{id}-{slug}`:

```bash frame="terminal" title="Worktree structure"
.worktrees/
  PRD-070-retry-webhook/          # worktree directory
  PRD-070.lock                    # advisory file lock (per PRD ID)
  PRD-071-dead-letter-queue/
  PRD-071.lock
```

The `ensure_worktree` builtin handles creation:

```python
BuiltIn(name="ensure_worktree")
# Equivalent to:
# git worktree add -b prd/PRD-070-retry-webhook \
#   .worktrees/PRD-070-retry-webhook main
```

## Lifecycle

<Steps>
1. `ensure_worktree` creates the worktree and branch from the base ref
2. Agent and ShellTasks execute with `cwd` set to the worktree path
3. Changes are committed and pushed from the worktree
4. PR is opened from the worktree branch
5. `cleanup_worktree` removes the worktree after success
</Steps>

The `cleanup_worktree` builtin runs `git worktree remove` to delete the directory. The branch persists on the remote for the open PR.

## File Locking

Each PRD run acquires an **advisory file lock** via the `filelock` library before creating or resuming its worktree:

```
.worktrees/PRD-070.lock
```

The lock is held for the **entire run** (from `ensure_worktree` through the final task) and released when the process exits. This prevents two concurrent `prd run` invocations for the same PRD from racing. Multiple PRDs run in parallel with no contention because each holds its own per-PRD lock.

## Stacked PRs

When a PRD has dependencies completed in the current run, the worktree can branch from a dependency's branch instead of the default base:

<Tabs>
  <TabItem label="Single Dep (Stacked)">
    If a PRD has **exactly 1 dependency** completed in the current run, its worktree branches from that dependency's branch:

    ```
    main
      +-- prd/PRD-068-http-client        (dep, completed this run)
            +-- prd/PRD-070-retry-webhook (branches from dep, not main)
    ```

    This produces a stacked PR -- PRD-070's diff is relative to PRD-068's branch. When PRD-068 merges, PRD-070 rebases onto main.
  </TabItem>
  <TabItem label="No Deps / All Merged">
    If a PRD has no unmerged dependencies from the current run, it branches from the default base (usually `main`):

    ```
    main
      +-- prd/PRD-004-standalone-task     (branches from main)
    ```
  </TabItem>
  <TabItem label="Multi-Dep (Unsupported)">
    If a PRD has **2 or more unmerged dependencies** from the current run, the worktree is skipped:

    ```
    # PRD-075 depends on both PRD-070 and PRD-071, both unmerged
    # Result: MultiDepUnsupported -- PRD-075 is skipped
    ```

    Multi-dependency stacking requires merge conflict resolution and is tracked as a future enhancement (PRD-552).
  </TabItem>
</Tabs>

<Aside type="caution">
Stacked PRs only apply when the dependency was completed in the **current execution run**. If a dependency was completed in a previous run and already merged, the new PRD branches from `main` as usual.
</Aside>

## Parallel Execution

Because each PRD gets its own worktree directory and branch, multiple PRDs can execute simultaneously without interference:

```
Agent 1: working in .worktrees/PRD-070-retry-webhook/
Agent 2: working in .worktrees/PRD-071-dead-letter-queue/
Agent 3: working in .worktrees/PRD-004-standalone-task/
```

Each agent sees a clean repository state. File changes in one worktree do not affect another. The only shared resource is the git object store, which is thread-safe in git.
