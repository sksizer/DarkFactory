# Compose rework feedback into agent prompt

## Summary

Create the prompt templates for the rework workflow. The key template composes the fetched PR review comments into a structured agent prompt that tells the agent what feedback to address, how to approach each comment, and what sentinel to emit when done.

## Requirements

1. Create a `workflows/rework/prompts/` directory with the rework-specific prompt templates.
2. The `role.md` template establishes the agent's role as a code reviewer addressing feedback.
3. The `task.md` template structures the rework task: original PRD context, git history reference, and the review feedback to address.
4. The `rework-feedback.md` template formats the list of `ReviewThread` objects into a readable markdown section with author, file path, line number, and comment body for each thread.
5. The prompt must instruct the agent to: read each comment, decide whether to address or push back, make edits, and prepare per-comment reply notes.
6. The prompt must include the sentinel contract (`PRD_EXECUTE_OK` / `PRD_EXECUTE_FAILED`).
7. The template system must be compatible with the existing `templates.py` composition logic.

## Technical Approach

### `workflows/rework/prompts/role.md`

```markdown
# Role

You are a senior engineer addressing PR review feedback. You have
already implemented the feature and opened a PR. A reviewer has left
comments. Your job is to address each comment by editing code or
explaining why you disagree.

## You MUST NOT

- Rewrite the entire implementation from scratch.
- Introduce new features beyond what the reviewer asked for.
- Delete or modify test files unless the reviewer specifically asked.

## Sentinel contract

Your **final line** of output must be exactly one of:

- `PRD_EXECUTE_OK: {{PRD_ID}}` — you addressed all feedback.
- `PRD_EXECUTE_FAILED: <reason>` — you could not complete the rework.
```

### `workflows/rework/prompts/task.md`

```markdown
# Rework: address PR review feedback for {{PRD_ID}}

## Original PRD
Read the PRD at `{{PRD_PATH}}` for context on what was implemented.

## Your previous work
The git history on this branch shows what you committed. Use
`git log` and `git diff main...HEAD` to see what's already done.

## Review feedback to address

{{REWORK_FEEDBACK}}

## Steps

1. Read each comment carefully.
2. For each comment, decide: address it (edit code) OR push back
   (note why you disagree).
3. Make the necessary code edits.
4. For each comment, prepare a one-line reply note:
   - "Addressed: <what you changed>"
   - "Disagree: <reason>"
   - "Already addressed in prior commit (please re-review)"
5. Stage all your changes — the harness commits and pushes.
```

### `workflows/rework/prompts/rework-feedback.md`

A Jinja-style template that iterates over the review threads:

```markdown
{% for thread in threads %}
### Comment by {{ thread.author }}{% if thread.path %} on `{{ thread.path }}`{% if thread.line %}:{{ thread.line }}{% endif %}{% endif %}

> {{ thread.body }}
{% if thread.replies %}

**Thread replies:**
{% for reply in thread.replies %}
> **{{ reply.author }}:** {{ reply.body }}
{% endfor %}
{% endif %}

---
{% endfor %}
```

### Integration with `templates.py`

The existing `compose_prompt` function in `templates.py` loads templates from `workflows/<name>/prompts/`. The rework templates must follow the same naming convention. The `REWORK_FEEDBACK` variable is rendered by formatting the `ReviewThread` list through the `rework-feedback.md` template before inserting it into `task.md`.

### `tests/test_rework_prompt.py`

- Test: Rendering with a single review thread produces expected markdown
- Test: Rendering with multiple threads (some with replies) produces expected output
- Test: Rendering with an issue-level comment (no path/line) handles it gracefully
- Test: Empty thread list produces a "No feedback to address" message
- Test: Template variables (PRD_ID, PRD_PATH) are substituted correctly
