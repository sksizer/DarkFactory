# Refactor execution loop to use candidate-selection strategy

## Summary

Factor the candidate-selection logic out of `execute_graph` so that both rooted mode (existing `graph_scope` + `actionable_order`) and queue mode (`discover_ready_queue`) can share the same run/reload/fail loop. The refactoring introduces a callable strategy that, given the current PRD set, returns the ordered list of candidate PRD IDs to consider.

## Requirements

1. Define a `CandidateStrategy` protocol or callable type: `(prds: dict[str, PRD]) -> list[str]` — returns ordered candidate IDs for the current reload cycle.
2. Refactor `execute_graph` to accept an optional `candidate_fn` parameter. When provided, it replaces the `graph_scope` + `actionable_order` pipeline.
3. The existing rooted-mode behavior must be preserved exactly — `execute_graph(root_id, ...)` with no `candidate_fn` behaves identically to today.
4. Create a `queue_candidate_fn(prd_dir, filters)` factory that returns a strategy calling `discover_ready_queue` on each reload.
5. `plan_execution` should similarly accept an optional candidate strategy for dry-run planning in queue mode.
6. Failure handling, sticky skips, blocked-id propagation, stacked-branch resolution, and event emission remain unchanged.

## Technical Approach

### File: `src/darkfactory/graph_execution.py`

#### Strategy type

```python
from typing import Protocol

class CandidateStrategy(Protocol):
    def __call__(self, prds: dict[str, PRD]) -> list[str]: ...
```

#### Rooted strategy (extract from existing code)

```python
def rooted_candidates(root_id: str) -> CandidateStrategy:
    def _candidates(prds: dict[str, PRD]) -> list[str]:
        scope = graph_scope(root_id, prds)
        return actionable_order(scope, prds)
    return _candidates
```

#### Queue strategy

```python
def queue_candidates(filters: QueueFilters) -> CandidateStrategy:
    def _candidates(prds: dict[str, PRD]) -> list[str]:
        return [p.id for p in discover_ready_queue(prds, filters)]
    return _candidates
```

#### Refactor `execute_graph`

Change the signature to:

```python
def execute_graph(
    root_id: str | None,
    prd_dir: Path,
    repo_root: Path,
    workflows: dict[str, Workflow],
    *,
    default_base: str,
    max_runs: int | None = None,
    candidate_fn: CandidateStrategy | None = None,
    ...
) -> ExecutionReport:
```

Inside the loop, replace:

```python
scope = graph_scope(root_id, prds)
order = actionable_order(scope, prds)
```

with:

```python
if candidate_fn is not None:
    order = candidate_fn(prds)
else:
    scope = graph_scope(root_id, prds)
    order = actionable_order(scope, prds)
```

The `root_id not in prds` guard becomes conditional on `root_id` being non-None.

#### Refactor `plan_execution`

Apply the same candidate-strategy pattern to `plan_execution` so that `prd run --all` dry-run uses the queue strategy.

## Acceptance Criteria

- [ ] AC-1: `execute_graph` accepts an optional `candidate_fn` parameter.
- [ ] AC-2: When `candidate_fn` is `None` and `root_id` is provided, behavior is identical to the pre-refactor code (rooted mode).
- [ ] AC-3: When `candidate_fn` is provided, the loop uses it instead of `graph_scope` + `actionable_order`.
- [ ] AC-4: `plan_execution` also accepts an optional candidate strategy for queue dry-run.
- [ ] AC-5: Failure isolation, sticky skips, stacked-branch resolution, and event emission are unaffected by the refactor.
- [ ] AC-6: All existing tests continue to pass without modification (or with minimal import-path changes).
