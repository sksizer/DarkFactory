# Add path filtering to ci.yml pull_request trigger

## Summary

Modify the existing `.github/workflows/ci.yml` so that the `pull_request` trigger only fires when source code, tests, or dependency files change. Push to `main` continues to run unconditionally as a safety net.

## Requirements

1. Add a `paths` filter to the `pull_request` trigger in `ci.yml` matching:
   - `src/**`
   - `tests/**`
   - `pyproject.toml`
   - `uv.lock`
   - `.github/workflows/ci.yml`
2. The `push` trigger for `main` must remain unconditional (no path filter).
3. The existing `test` job content (`uv sync`, `pytest`, `mypy`) is unchanged.

## Technical Approach

### `.github/workflows/ci.yml`

Update the `on:` block to:

```yaml
on:
  push:
    branches: [main]
  pull_request:
    paths:
      - 'src/**'
      - 'tests/**'
      - 'pyproject.toml'
      - 'uv.lock'
      - '.github/workflows/ci.yml'
```

No changes to the `jobs:` section.

## Acceptance Criteria

- [ ] AC-1: A PR that only touches `.darkfactory/prds/*.md` files does NOT trigger `ci.yml`.
- [ ] AC-2: A PR that touches `src/**` or `tests/**` triggers `ci.yml`.
- [ ] AC-3: Push to `main` triggers `ci.yml` unconditionally.
- [ ] AC-4: The `test` job steps are unchanged.
