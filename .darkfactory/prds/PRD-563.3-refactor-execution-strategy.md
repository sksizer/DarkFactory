# Refactor execution loop with candidate-strategy pattern

## Summary

Factor the candidate-selection logic out of `execute_graph` so both
rooted mode and queue mode share the same run/re-load/fail/stacking
loop. The execution loop itself is identical between modes — only the
way the initial candidate set is determined differs.

This is the key architectural change that enables queue mode without
duplicating the execution, failure-handling, and stacking logic.

## Requirements

1. Extract a `CandidateStrategy` protocol (or callable type) that,
   given the current PRD set, returns an ordered list of candidate
   PRD IDs to consider running.
2. Implement `RootedStrategy` — wraps the existing `graph_scope` +
   `actionable_order` logic for a given root PRD ID.
3. Implement `QueueStrategy` — wraps `discover_ready_queue` from
   PRD-563.1, returning all ready PRDs repo-wide.
4. Refactor `execute_graph` to accept a strategy instead of (or in
   addition to) a `root_id`. The existing `root_id`-based signature
   must remain as a convenience that creates a `RootedStrategy`
   internally, preserving backward compatibility.
5. Refactor `plan_execution` similarly to support both modes for
   dry-run output.
6. Failure handling, stacking, re-load, and `--max-runs` semantics
   are unchanged — they stay in the shared loop.
7. In queue mode, the re-load step also re-runs discovery, so PRDs
   that become ready mid-run (because a dependency just completed)
   are picked up on the next iteration.

## Technical Approach

### File: `src/darkfactory/graph_execution.py`

Define the strategy protocol:

```python
from typing import Protocol

class CandidateStrategy(Protocol):
    def candidates(self, prds: dict[str, PRD]) -> list[str]:
        """Return ordered candidate PRD IDs for this iteration."""
        ...

class RootedStrategy:
    def __init__(self, root_id: str) -> None:
        self.root_id = root_id

    def candidates(self, prds: dict[str, PRD]) -> list[str]:
        scope = graph_scope(self.root_id, prds)
        return actionable_order(scope, prds)

class QueueStrategy:
    def __init__(self, filters: QueueFilters) -> None:
        self.filters = filters

    def candidates(self, prds: dict[str, PRD]) -> list[str]:
        return [p.id for p in discover_ready_queue(prds, self.filters)]
```

Refactor `execute_graph` to accept `strategy: CandidateStrategy`:

```python
def execute_graph(
    strategy: CandidateStrategy,
    prd_dir: Path,
    repo_root: Path,
    workflows: dict[str, Workflow],
    *,
    default_base: str,
    max_runs: int | None = None,
    ...
) -> ExecutionReport:
```

Add a backward-compatible wrapper or keep `root_id` as an optional
parameter that creates `RootedStrategy` internally. The existing test
suite and `cmd_run` call sites must continue to work without changes.

Similarly refactor `plan_execution` to accept a strategy.

## Acceptance Criteria

- [ ] AC-1: `CandidateStrategy` protocol is defined with a `candidates`
      method.
- [ ] AC-2: `RootedStrategy` reproduces the existing rooted traversal
      behavior exactly.
- [ ] AC-3: `QueueStrategy` uses `discover_ready_queue` to find
      candidates across the entire repo.
- [ ] AC-4: `execute_graph` accepts a strategy and the existing
      `root_id`-based call pattern still works (backward compatible).
- [ ] AC-5: `plan_execution` supports both strategies for dry-run.
- [ ] AC-6: Re-load between runs re-invokes the strategy, picking up
      newly-ready PRDs in queue mode.
- [ ] AC-7: Failure handling, stacking, and `--max-runs` semantics
      are unchanged from PRD-220.
