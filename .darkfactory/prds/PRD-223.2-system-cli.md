# prd system CLI subcommand (list, describe, run)

## Summary

Add the `prd system` CLI subcommand group with three sub-subcommands: `list`, `describe <name>`, and `run <name> [--execute] [--target PRD-X]`. This gives users a clear interface to discover and invoke system operations, separate from the PRD-bound `prd run` / `prd plan` commands.

Also implement `run_system_operation()` ‚Äî the dispatch loop for system operations, analogous to `run_workflow()` in `runner.py` but operating on a `SystemContext` instead of `ExecutionContext`.

## Requirements

1. `prd system list` ‚Äî load all operations from `.darkfactory/operations/`, display name + description for each, exit 0.
2. `prd system describe <name>` ‚Äî show the operation's name, description, task list (with types), `creates_pr`, and `requires_clean_main` fields.
3. `prd system run <name>` ‚Äî dry-run by default. Load the operation, build a `SystemContext`, dispatch each task. Print the report lines and targets at the end.
4. `prd system run <name> --execute` ‚Äî same as above but with `dry_run=False`. Actually mutate files, commit, push, create PR if `creates_pr=True`.
5. `prd system run <name> --target PRD-X` ‚Äî pass `target_prd` to the context. Refuse if the operation has `accepts_target=False`.
6. `run_system_operation(operation, ctx, model_override=None) -> RunResult` ‚Äî dispatch loop that walks `operation.tasks`, dispatching each to the same handlers as `run_workflow` (BuiltIn, AgentTask, ShellTask). The key difference: builtins receive a `SystemContext` instead of `ExecutionContext`.
7. The system runner must respect the process lock (PRD-217) when `--execute` is used.
8. System ops are NOT shown in `prd next` / `prd plan` / `prd run` ‚Äî clean separation.
