# SystemOperation + SystemContext core abstractions

## Summary

Introduce the `SystemOperation` and `SystemContext` dataclasses in a new `src/darkfactory/system.py` module. These are the sibling concepts to `Workflow` and `ExecutionContext` — they reuse the same `Task` types (`BuiltIn`, `AgentTask`, `ShellTask`) but target a different execution shape: no single PRD, optionally read-only, with output that's either a report or a single batched PR.

Also extend the existing `loader.py` with a `load_operations()` function that discovers operations from `.darkfactory/operations/` using the same pattern as `load_workflows()`.

## Requirements

1. `SystemOperation` dataclass with fields: `name`, `description`, `tasks: list[Task]`, `requires_clean_main: bool`, `creates_pr: bool`, `pr_title: str | None`, `pr_body: str | None`, `accepts_target: bool` (for operations like planning that take a single PRD as input).
2. `SystemContext` dataclass with fields: `repo_root: Path`, `prds: dict[str, PRD]`, `operation: SystemOperation`, `cwd: Path`, `dry_run: bool`, `logger`, `targets: list[str]`, `report: list[str]`, `pr_url: str | None`, `_shared_state: dict[str, Any]`, `target_prd: str | None` (for `accepts_target` operations).
3. `SystemContext.format_string(template)` — analogous to `ExecutionContext.format_string` but with system-op-relevant placeholders (`{operation_name}`, `{target_count}`, `{target_prd}` if set).
4. `load_operations(operations_dir: Path) -> dict[str, SystemOperation]` — mirrors the existing `load_workflows` pattern: scan subdirectories for `operation.py` modules exporting an `operation` attribute, set `operation_dir`, handle import errors gracefully, raise on duplicate names.
5. The module must be pure data + loader logic — no I/O beyond the loader's `importlib` calls.

## Technical Approach

**New file**: `src/darkfactory/system.py`

```python
from __future__ import annotations

import logging
from dataclasses import dataclass, field
from pathlib import Path
from typing import TYPE_CHECKING, Any

from .workflow import Task

if TYPE_CHECKING:
    from .prd import PRD


@dataclass
class SystemOperation:
    name: str
    description: str
    tasks: list[Task]
    requires_clean_main: bool = True
    creates_pr: bool = False
    pr_title: str | None = None
    pr_body: str | None = None
    accepts_target: bool = False
    operation_dir: Path | None = None


@dataclass
class SystemContext:
    repo_root: Path
    prds: dict[str, "PRD"]
    operation: SystemOperation
    cwd: Path
    dry_run: bool = True
    logger: logging.Logger = field(
        default_factory=lambda: logging.getLogger("darkfactory.system")
    )
    targets: list[str] = field(default_factory=list)
    report: list[str] = field(default_factory=list)
    pr_url: str | None = None
    target_prd: str | None = None
    _shared_state: dict[str, Any] = field(default_factory=dict)

    def format_string(self, template: str) -> str:
        return template.format(
            operation_name=self.operation.name,
            target_count=len(self.targets),
            target_prd=self.target_prd or "",
        )
```

**Modified file**: `src/darkfactory/loader.py` — add `load_operations()` function using the same importlib pattern as `load_workflows`, but scanning for `operation.py` modules exporting an `operation` attribute of type `SystemOperation`.

**New file**: `tests/test_system.py` — unit tests for dataclass construction, `format_string`, and `load_operations` discovery (using tmp_path fixtures).

## Acceptance Criteria

- [ ] AC-1: `SystemOperation` and `SystemContext` are importable from `darkfactory.system`
- [ ] AC-2: `SystemContext.format_string` expands `{operation_name}`, `{target_count}`, `{target_prd}` correctly
- [ ] AC-3: `load_operations` discovers operations from a directory of `operation.py` modules
- [ ] AC-4: `load_operations` raises `ValueError` on duplicate operation names
- [ ] AC-5: `load_operations` logs and skips modules with import errors
- [ ] AC-6: `load_operations` returns empty dict for nonexistent directory
