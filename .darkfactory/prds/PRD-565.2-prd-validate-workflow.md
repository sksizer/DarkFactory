# Create dedicated prd-validate.yml workflow

## Summary

Create a new `.github/workflows/prd-validate.yml` workflow that runs `uv run prd validate` on every pull request and push to `main`, with no path filter. This ensures PRD structural validation (schema, references, cycles) is always enforced in CI regardless of which files changed.

## Requirements

1. Create `.github/workflows/prd-validate.yml` triggered on `pull_request` (all paths) and `push` to `main`.
2. The workflow installs the project with `uv sync` and runs `uv run prd validate`.
3. Non-zero exit from `prd validate` must fail the workflow.
4. Use `actions/checkout@v4` and `astral-sh/setup-uv@v3` with caching enabled.
5. Pin Python to 3.12 to match the main CI workflow.

## Technical Approach

### `.github/workflows/prd-validate.yml`

```yaml
name: PRD Validation
on:
  push:
    branches: [main]
  pull_request:
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v3
        with:
          enable-cache: true
      - run: uv python install 3.12
      - run: uv sync
      - run: uv run prd validate
```

The workflow is intentionally simple. `uv run prd validate` already exits non-zero on validation failures, so no additional error handling is needed.

## Acceptance Criteria

- [ ] AC-1: `prd-validate.yml` triggers on every `pull_request` regardless of paths changed.
- [ ] AC-2: `prd-validate.yml` triggers on push to `main`.
- [ ] AC-3: The workflow runs `uv run prd validate` and fails the check on non-zero exit.
- [ ] AC-4: A PRD with a broken reference causes the validation workflow to fail.
