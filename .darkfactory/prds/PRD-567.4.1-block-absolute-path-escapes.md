# Block absolute-path escapes in invoke.py

## Summary

The existing containment rules (`--disallowed-tools "Edit(../)"`) only block relative-path traversal. An agent can bypass them using absolute paths like `Write("/Users/sksizer2/Developer/DarkFactory/.darkfactory/prds/file.md")`. This task computes the repo root at invocation time and adds `--disallowed-tools` entries that block write operations targeting the repo root (while still allowing writes within the worktree, which is a subdirectory).

## Requirements

1. Compute `repo_root` absolute path at invocation time (already available via `cwd` parameter context).
2. Add `--disallowed-tools` entries that block `Write`, `Edit`, and `Read` operations using absolute paths outside the worktree.
3. The containment must allow operations within the worktree directory (which is under `.worktrees/` inside the repo root).
4. Must handle edge cases: worktree path is a subdirectory of repo root, so a naive "block repo root" rule would also block worktree writes.
5. Unit tests covering: absolute path to repo root blocked, absolute path within worktree allowed, relative `../` still blocked.

## Technical approach

Claude Code's `--disallowed-tools` patterns are prefix-matched. The strategy:

1. Block all absolute-path writes by denying `Write(/*)`, `Edit(/*)`.
2. Since `--allowed-tools` takes precedence over `--disallowed-tools` for the same tool, and file tools use the `cwd` to resolve relative paths, relative-path operations within the worktree will work.

Alternatively, if Claude Code doesn't support this precedence model:

1. Compute the worktree absolute path.
2. Compute the repo root absolute path.
3. Add deny rules for the repo root but NOT for the worktree subdirectory path.

This requires testing to determine how Claude Code resolves allow/deny conflicts for file tools. The implementation should include a discovery step to verify the exact semantics.

```python
# In invoke.py, after existing --disallowed-tools:
# Block absolute-path escapes. Since the agent works in a worktree
# (cwd), all legitimate file operations use relative paths. Blocking
# absolute paths prevents escaping to the main repo.
"--disallowed-tools", "Write(/*)",
"--disallowed-tools", "Edit(/*)",
```

If this approach blocks legitimate worktree operations (because Claude Code resolves relative paths to absolute before checking), fall back to the explicit deny approach:

```python
repo_abs = str(repo_root.resolve())
# Block the repo root specifically, not all absolute paths
"--disallowed-tools", f"Write({repo_abs}/*)",
"--disallowed-tools", f"Edit({repo_abs}/*)",
```

## Open questions

- [ ] How does Claude Code resolve `--disallowed-tools` patterns for file tools — are they matched against the raw argument or the resolved absolute path? This determines whether `Write(/*)` blocks relative-path operations too.
- [ ] Does `--allowed-tools` take precedence over `--disallowed-tools` for the same tool, or is deny-wins the policy?

## Acceptance criteria

- [ ] AC-1: An agent in a worktree cannot `Write` to the repo root using an absolute path.
- [ ] AC-2: An agent in a worktree cannot `Edit` files in the repo root using an absolute path.
- [ ] AC-3: Relative-path `Write` and `Edit` operations within the worktree continue to work.
- [ ] AC-4: The existing `../` deny rules are preserved (belt and suspenders).
- [ ] AC-5: Unit tests verify the constructed command includes the new containment rules.
- [ ] AC-6: `just test`, `just lint`, `just typecheck` all pass.
