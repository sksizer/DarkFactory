# Rework loop detection

## Summary

A safety check that detects when `prd rework PRD-X` produces no changes. If the agent runs but stages nothing, this indicates a potential loop: the agent thinks comments are already addressed, disagrees with the reviewer, or is confused. The harness must flag this loudly rather than silently no-op.

## Requirements

1. After the agent runs in the rework workflow, check whether any files were modified (via `git diff --cached` or `git status`).
2. If no changes were staged, flag the situation with a clear warning message.
3. Track consecutive no-change rework attempts per PRD in `.darkfactory/state/rework-guard.json`.
4. After N consecutive no-change attempts (default 2), escalate from warning to error and refuse further automatic rework for that PRD until the user manually intervenes.
5. Reset the counter when a rework run successfully produces changes.
6. Integrate with the rework-watch daemon (PRD-225.6) so the watcher respects the guard state.

## Technical Approach

### `src/darkfactory/rework_guard.py`

```python
@dataclass
class GuardState:
    no_change_counts: dict[str, int]  # prd_id -> consecutive no-change count

def check_rework_result(
    prd_id: str,
    has_changes: bool,
    state: GuardState,
    max_no_change: int = 2,
) -> tuple[GuardState, str | None]:
    """Check rework result and return updated state + optional error message."""
    if has_changes:
        state.no_change_counts.pop(prd_id, None)
        return state, None

    count = state.no_change_counts.get(prd_id, 0) + 1
    state.no_change_counts[prd_id] = count

    if count >= max_no_change:
        return state, (
            f"BLOCKED: {prd_id} has had {count} consecutive rework runs with no changes. "
            f"The agent may be unable to address the feedback. Manual intervention required."
        )
    return state, (
        f"WARNING: Rework of {prd_id} produced no changes. "
        f"The agent may think comments are already addressed or may disagree. "
        f"({count}/{max_no_change} before blocking)"
    )

def load_guard_state() -> GuardState:
    """Load guard state from .darkfactory/state/rework-guard.json."""
    ...

def save_guard_state(state: GuardState) -> None:
    """Save guard state."""
    ...
```

### Integration in rework workflow

After the agent step and before commit:

```python
has_changes = bool(subprocess.run(
    ["git", "diff", "--cached", "--quiet"],
    cwd=ctx.worktree_path,
).returncode)

guard_state = load_guard_state()
guard_state, message = check_rework_result(ctx.prd.id, has_changes, guard_state)
save_guard_state(guard_state)

if message and message.startswith("BLOCKED"):
    console.print(f"[red]{message}[/red]")
    raise SystemExit(1)
elif message:
    console.print(f"[yellow]{message}[/yellow]")

if not has_changes:
    console.print(f"[yellow]No changes to commit for {ctx.prd.id}[/yellow]")
    return  # Skip commit and push
```

### `tests/test_rework_guard.py`

- Test: Rework with changes → counter reset, no message
- Test: First no-change rework → warning with count

- Test: Changes after a no-change run → counter resets to 0
- Test: State file round-trips correctly
- Test: Different PRDs have independent counters

## Acceptance Criteria

- [ ] AC-1: A rework run that produces no changes emits a clear warning.
- [ ] AC-2: After N consecutive no-change runs, the harness blocks further automatic rework.
- [ ] AC-3: Producing changes resets the consecutive no-change counter.
- [ ] AC-4: The guard state persists across invocations via the state file.
- [ ] AC-5: The rework-watch daemon respects the guard and skips blocked PRDs.
