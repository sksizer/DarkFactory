# Core WorkflowTemplate abstraction

## Summary

Introduce the `WorkflowTemplate` dataclass and its `.compose()` method. A template defines required opening tasks, a constrained middle slot, and required closing tasks. `.compose()` concatenates `[*open, *middle, *close]` into a `Workflow`, validating the middle against `middle_kinds` and `middle_required` constraints. The composed workflow exposes a `template_name` field so the runner can assert provenance.

This is the foundational primitive that all other PRD-227 tasks build on.

## Requirements

1. Define a `WorkflowTemplate` dataclass with fields:
   - `name: str` — unique template identifier
   - `description: str`
   - `open: list[BuiltIn]` — required opening tasks, run in order
   - `close: list[BuiltIn]` — required closing tasks, run in order
   - `middle_kinds: list[type]` — allowed task types in the middle (e.g. `[AgentTask, ShellTask]`)
   - `middle_required: dict[type, tuple[int, int | None]]` — minimum/maximum count constraints per task type
2. Implement `.compose(name, description, applies_to, priority, middle)` that:
   - Validates every task in `middle` is an instance of one of `middle_kinds`; raises `TemplateViolation` if not
   - Validates `middle_required` count constraints; raises `TemplateViolation` if violated
   - Returns a `Workflow` with tasks `[*self.open, *middle, *self.close]`
   - Sets `template_name` on the resulting `Workflow`
3. Add a `template_name: str | None = None` field to `Workflow`.
4. Define `TemplateViolation` as a clear, user-facing exception with a message explaining what constraint was violated.
5. Handle the module naming collision: the existing `templates.py` handles prompt templates — either rename it to `prompts.py` or name the new module `workflow_templates.py`. Choose whichever minimizes churn.

## Technical Approach

### New module: `src/darkfactory/templates.py`

```python
from dataclasses import dataclass, field
from typing import Callable

from .workflow import Workflow, BuiltIn, AgentTask, ShellTask


class TemplateViolation(Exception):
    """Raised when a composed middle violates the template's constraints."""


@dataclass(frozen=True)
class WorkflowTemplate:
    name: str
    description: str
    open: list[BuiltIn]
    close: list[BuiltIn]
    middle_kinds: list[type] = field(default_factory=list)
    middle_required: dict[type, tuple[int, int | None]] = field(default_factory=dict)

    def compose(
        self,
        name: str,
        description: str,
        applies_to: Callable,
        priority: int,
        middle: list,
    ) -> Workflow:
        self._validate_middle(middle)
        return Workflow(
            name=name,
            description=description,
            applies_to=applies_to,
            priority=priority,
            tasks=[*self.open, *middle, *self.close],
            template_name=self.name,
        )

    def _validate_middle(self, middle: list) -> None:
        # Check kinds
        for task in middle:
            if not isinstance(task, tuple(self.middle_kinds)):
                raise TemplateViolation(
                    f"Task {task!r} is not an allowed middle kind "
                    f"(allowed: {[k.__name__ for k in self.middle_kinds]})"
                )
        # Check required counts
        for kind, (min_count, max_count) in self.middle_required.items():
            actual = sum(1 for t in middle if isinstance(t, kind))
            if actual < min_count:
                raise TemplateViolation(
                    f"Template requires at least {min_count} {kind.__name__} "
                    f"in the middle, got {actual}"
                )
            if max_count is not None and actual > max_count:
                raise TemplateViolation(
                    f"Template allows at most {max_count} {kind.__name__} "
                    f"in the middle, got {actual}"
                )
```

- [ ] AC-1: `WorkflowTemplate.compose(middle=[...])` produces a `Workflow` whose tasks are `[*open, *middle, *close]`.
- [ ] AC-2: Composing with a middle that violates `middle_kinds` raises `TemplateViolation` with a clear message.
- [ ] AC-3: Composing with a middle that violates `middle_required` raises `TemplateViolation` with a clear message.
- [ ] AC-4: The composed workflow exposes `template_name` matching the template's `name`.
- [ ] AC-5: Existing `Workflow(tasks=[...])` construction still works with `template_name` defaulting to `None`.
- [ ] AC-6: Test coverage for all validation paths (valid, invalid kind, too few, too many).
