# Harden role/task prompts against denial retry loops

## Summary

The current role prompts say "Do not run `git commit`" but agents still attempt it — often in compound commands like `git add && git commit`. The prompt doesn't explain *why* the command will be blocked or that retrying is pointless. Agents interpret the denial as a recoverable error and waste turns splitting the command and retrying.

This task adds concrete examples and explicit "do not retry denied commands" instructions to all workflow prompts.

## Requirements

1. Add a "Blocked commands" section to every `role.md` that lists specific commands the agent must not attempt, with a brief explanation of why each is blocked.
2. Add a "Permission denials" section explaining that if any command returns "requires approval" or "disallowed", the agent must not retry it — move on immediately.
3. In `task.md`, update the "Stage your changes" step to explicitly use separate `git add` commands (not compound `&&` chains).
4. Apply changes to all workflow prompt directories: `default/`, `task/`, `planning/`.

## Technical approach

Add to each `role.md` after the existing "You MUST NOT" section:

~~~markdown
## Blocked commands — do not attempt

These commands are explicitly blocked by the harness. They will be
instantly rejected. Do NOT attempt them, and do NOT retry if you
see a denial:

- `git commit` (any form, any flags) — blocked, harness commits after you
- `git push` — blocked, harness pushes after you
- `gh pr create` / `gh pr merge` — blocked, harness owns PR lifecycle
- `git checkout` / `git switch` / `git branch` — blocked, harness owns branching
- Compound commands like `git add && git commit` — the `git commit` part will be rejected

If you see "requires approval" or "disallowed" for any command,
do NOT retry it. The denial is permanent for this session. Move on
to your next step.
~~~

Update `task.md` step 4 to replace the compound example:

~~~markdown
### 4. Stage your changes

Stage each file individually or use `git add -A`:

```bash
git add -A
```

Then verify:

```bash
git status
git diff --cached
```

Do NOT chain `git add` with `git commit` using `&&`. The commit
will be rejected and the entire compound command will fail.
~~~
