# Make project-level workflow loading failures loud

## Summary

`loader.py:113-116` catches `Exception` during workflow loading and logs a warning. If a project-level workflow has a syntax error or import failure, it is silently skipped. The `assign_workflow` function then picks the highest-priority remaining workflow -- which may be a built-in with completely different behavior. This can cause a planning workflow to silently fall back to a default implementation workflow.

## Technical approach

Differentiate between loading layers:

1. **Built-in workflows** (in `darkfactory/workflows/`): Continue catching exceptions and logging warnings. These are the harness's own code and should always load; a failure here is a bug but shouldn't crash the user.

2. **Project-level workflows** (in `.darkfactory/workflows/`): Re-raise exceptions. If the user's project has a workflow with a bug, they need to know immediately, not have it silently replaced.

3. **User-level workflows** (in `~/.config/darkfactory/workflows/`): Log a warning but at `WARNING` level (not `DEBUG`). These are cross-project and may legitimately fail in some project contexts.

### Code location

`loader.py:113-116` -- the `except Exception` handler in the workflow loading loop. The `load_workflows` function is called with different directory paths for each layer. Add a `strict=False` parameter that defaults to `False` for backwards compatibility. `registry.py` (which calls `load_workflows`) should pass `strict=True` for the project layer.

## Acceptance criteria

- [ ] AC-1: A broken project-level workflow raises an error, not a silent fallback
- [ ] AC-2: A broken built-in workflow logs a warning and continues
- [ ] AC-3: The error message includes the workflow file path and the underlying exception
- [ ] AC-4: Existing tests pass (they use built-in workflow isolation fixtures)
- [ ] AC-5: New test covers a broken project workflow raising an error
