# Structured failure context in event logs

## Summary

When a task fails, the JSONL event log records `success: false` with a brief `detail` string, but omits stderr and structured diagnostic information. The PRD-227.3 `gh pr create` failure logged only `exit status 1` — no stderr, no command output, nothing to diagnose the issue without re-running.

This task enriches failure events with stderr, stdout (truncated), and structured error context so post-mortem analysis doesn't require re-running the failed task.

## Requirements

1. When a `ShellTask` fails, include `stderr` (truncated to 2000 chars) and `exit_code` in the `task_finish` event.
2. When a `BuiltIn` task raises, include the exception message and type in the `task_finish` event.
3. When an `AgentTask` fails (sentinel `PRD_EXECUTE_FAILED` or timeout), include the failure reason and last 2000 chars of agent output in the `task_finish` event.
4. Add an optional `error_context` dict field to `task_finish` events for structured data.
5. Existing event consumers must not break — `error_context` is additive.

## Technical approach

### Event schema extension

```python
# In the task_finish event emission (runner.py):
#
# Note: ShellTask results use subprocess's `returncode` attribute,
# while AgentTask returns an InvokeResult with `exit_code`. Normalize
# to a single `exit_code` field in the event before emitting.
event = {
    "type": "task_finish",
    "task": task.name,
    "success": False,
    "detail": failure_reason,
    "duration_ms": elapsed_ms,
    # NEW: structured error context
    "error_context": {
        "stderr": result.stderr[-2000:] if hasattr(result, 'stderr') and result.stderr else None,
        "stdout_tail": result.stdout[-2000:] if hasattr(result, 'stdout') and result.stdout else None,
        # ShellTask: proc.returncode; AgentTask: invoke_result.exit_code
        "exit_code": getattr(result, 'exit_code', None) or getattr(result, 'returncode', None),
        "exception_type": type(exc).__name__ if exc else None,
        "exception_message": str(exc)[:500] if exc else None,
    },
}
```

### Runner changes

In each task dispatch path (BuiltIn, AgentTask, ShellTask), capture the failure context and pass it to the event writer. The event writer already accepts arbitrary extra fields via `**kwargs`, so no changes to `event_log.py` are needed — just pass the `error_context` dict.

## Acceptance criteria

- [ ] AC-1: `task_finish` events for failed `ShellTask`s include `error_context.stderr` and `error_context.exit_code`.
- [ ] AC-2: `task_finish` events for failed `BuiltIn` tasks include `error_context.exception_type` and `error_context.exception_message`.
- [ ] AC-3: `task_finish` events for failed `AgentTask`s include `error_context.stdout_tail` with the last 2000 chars of agent output.
- [ ] AC-4: `error_context` is omitted (or null) for successful task events — no noise.
- [ ] AC-5: Existing event log consumers (jq queries, event replay) are not broken by the new field.
- [ ] AC-6: `just test`, `just lint`, `just typecheck` all pass.
