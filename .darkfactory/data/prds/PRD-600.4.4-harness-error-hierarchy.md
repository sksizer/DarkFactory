# Introduce HarnessError exception hierarchy

## Summary

All error paths use `raise SystemExit("message")`. While correct for a CLI tool, `SystemExit` cannot be caught and translated by a server process. Introduce a structured exception hierarchy that the CLI's `main()` catches and converts to `SystemExit`, while a future server could catch and convert to HTTP responses.

## Technical approach

1. Create `python/darkfactory/errors.py`:
   ```python
   class HarnessError(Exception):
       """Base exception for all DarkFactory errors."""

   class UserError(HarnessError):
       """User input or configuration error (bad PRD ID, invalid flags)."""

   class ExecutionError(HarnessError):
       """Runtime execution failure (git failed, network error)."""

   class ValidationError(HarnessError):
       """PRD validation failure (cycles, missing deps)."""
   ```

2. In `cli/main.py`, wrap `func(args)` in:
   ```python
   try:
       return func(args)
   except HarnessError as e:
       print(str(e), file=sys.stderr)
       return 1
   ```

3. Gradually replace `raise SystemExit("message")` with `raise UserError("message")` etc. across CLI handlers. This can be done incrementally -- both `SystemExit` and `HarnessError` work in parallel during migration.

**Important:** The adversarial review notes this is at least 4-6 hours of work (30+ SystemExit sites), not the "~1 hour" estimated by the frontend reviewer. Plan accordingly.

## Acceptance criteria

- [ ] AC-1: `errors.py` defines `HarnessError`, `UserError`, `ExecutionError`, `ValidationError`
- [ ] AC-2: `main()` catches `HarnessError` and exits with code 1
- [ ] AC-3: At least the 5 most common error paths use the new hierarchy
- [ ] AC-4: Existing error messages and exit codes are preserved
- [ ] AC-5: `mypy --strict` passes
- [ ] AC-6: `--json` mode emits structured error objects for `HarnessError` subclasses
