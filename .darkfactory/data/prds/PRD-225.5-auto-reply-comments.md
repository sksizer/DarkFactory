# Auto-post replies to addressed PR comments

## Summary

After the rework agent finishes, optionally post replies on each PR comment that was addressed. The agent's per-comment reply notes are extracted from its output and posted via `gh` with a `[harness]` prefix so reviewers can distinguish bot replies from human ones. This is opt-in via the `--reply-to-comments` flag.

## Current State

Blocked on PRD-225.4 (rework workflow definition). No implementation exists yet.

## Known Risks

The agent output parsing approach (matching `Addressed:` lines positionally to threads) is fragile. A more robust approach might be to have the agent emit structured output (JSON) with explicit thread ID references, or to use the agent's diff to infer which comments were addressed by file/line proximity.

## Requirements

1. Extract per-comment reply notes from the agent's output. The agent is instructed to produce structured markers indicating how each comment was addressed.
2. Parse the agent output to match reply notes to their corresponding review threads (by thread ID or structured reference — avoid purely positional matching).
3. Post replies using `gh api` to comment on the specific review thread.
4. Prefix each reply with `[harness] addressed in {commit_sha}: ` so it's clearly identifiable as a bot reply.
5. Only execute when `--reply-to-comments` is passed — this is opt-in.
6. Handle failures gracefully: if posting a reply fails (rate limit, permission), log a warning but don't fail the rework run.

## Technical Approach

### Reply posting (`src/darkfactory/pr_comments.py`)

Add reply functions to the existing module:

```python
@dataclass
class CommentReply:
    thread_id: str
    body: str

def post_comment_replies(
    pr_number: int,
    replies: list[CommentReply],
    commit_sha: str,
    repo_root: Path,
) -> list[tuple[str, bool]]:
    """Post replies to PR comment threads. Returns (thread_id, success) pairs."""
    ...
```

### Agent output parsing

The parser needs to be more robust than positional matching. Options:
- Require the agent to emit JSON-structured reply blocks with explicit thread references
- Match by file path + line proximity between comment location and agent changes
- Use a secondary agent pass to classify which comments were addressed (expensive but reliable)

Decision deferred until PRD-225.4 is implemented and the agent output format is established.

### Integration in rework workflow

The reply-posting step runs as a BuiltIn after commit and push in the rework workflow's close sequence, only if `ctx.reply_to_comments` is True. Failed posts log warnings but don't fail the run.

## Acceptance Criteria

- [ ] `--reply-to-comments` flag triggers reply posting after successful rework
- [ ] Each reply is prefixed with `[harness] addressed in {commit_sha}:`
- [ ] Failed reply posts log a warning but do not fail the rework run
- [ ] Without `--reply-to-comments`, no replies are posted
- [ ] Agent reply notes are correctly parsed and matched to review threads
- [ ] Tests cover: reply posting, failure handling, opt-in gating, parsing
