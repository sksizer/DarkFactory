# Add crash recovery to `_create_reconcile_pr`

## Summary

`_create_reconcile_pr` in `cli/__init__.py` executes 6 sequential destructive git/gh subprocess calls (`git branch -D`, `git checkout -b`, `git add`, `git commit`, `git push`, `gh pr create`) with `check=True` and zero rollback logic. A failure at any point after `git checkout -b` leaves the user stranded on a non-main branch with partial changes and no indication of what to do.

## Technical approach

1. Record the current branch at function entry: `original_branch = subprocess.run(["git", "rev-parse", "--abbrev-ref", "HEAD"], ...).stdout.strip()`
2. Wrap the entire subprocess chain in `try/finally`
3. In the `finally` block, check if we're still on the reconcile branch and `git checkout` back to `original_branch`
4. If the reconcile branch has uncommitted changes from a partial failure, reset them before switching: `git checkout -- .`
5. Log a clear message about what happened and what the user should do

### Code location

`cli/__init__.py:1441-1478` (line numbers approximate -- look for `_create_reconcile_pr` function).

The destructive sequence is:
- Line ~1448: `git branch -D prd/reconcile-status` (deletes existing branch)
- Line ~1452: `git checkout -b prd/reconcile-status` (switches HEAD away from main)
- Lines ~1456-1478: `git add`, `git commit`, `git push`, `gh pr create`

### Edge cases

- If `git branch -D` fails (branch doesn't exist) -- this is OK, the subsequent `checkout -b` will work. The current `check=True` will abort unnecessarily. Change to `check=False` for the delete and check return code.
- If `git push` fails (network error) -- the commit is local, user is on reconcile branch. The finally block should switch back and leave the branch for manual cleanup.
- If the user was already on a non-main branch when reconcile started -- respect that and return to it.
