# File operation permissions and workflow specialization

## Summary

PRD-549.12 is the only task-level failure in 36 analyzed transcripts — it failed because the agent couldn't delete a file despite the task workflow including `Bash(git rm:*)`. The agent likely used `rm` instead of `git rm`, or Claude Code's sandbox applied an additional deletion guard.

This feature fixes deletion permissions in the task workflow and creates a new `refactor` workflow with broader file-operation permissions for PRDs that primarily involve reorganizing, moving, or deleting code.

## Acceptance criteria

- [ ] AC-1: The task workflow allows both `git rm` and direct `rm` for file deletion.
- [ ] AC-2: A dedicated `refactor` workflow exists with broad file-operation permissions.

## Assessment (2026-04-11)

- **Value**: 3/5 — PRD-549.12 (the only task-level failure in the
  analyzed dataset) would have succeeded with this. Low
  frequency but unblocks a real class of refactor work.
- **Effort**: s — add `Bash(rm:*)` and `Bash(mv:*)` to task
  workflow allowlist (567.3.1 = xs), create `workflows/refactor/`
  with wider permissions (567.3.2 = s).
- **Current state**: greenfield on both. State survey confirms
  no `refactor` workflow directory exists.
- **Gaps**: see the two child assessments.
- **Recommendation**: do-next — ships naturally after 567.1 and
  567.2. Also pairs with PRD-553 (child creation permission)
