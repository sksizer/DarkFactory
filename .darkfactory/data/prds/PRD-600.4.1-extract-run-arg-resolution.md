# Extract `cmd_run` argument resolution to standalone function

## Summary

`cmd_run` in the CLI layer mixes argument parsing with orchestration logic: loading PRDs, loading workflows, resolving the base ref, checking runnability, and wiring up event sinks. This logic is needed by any frontend that wants to execute a workflow. Extract the argument resolution and setup into a callable function that accepts typed parameters rather than `argparse.Namespace`.

## Technical approach

After PRD-556.16 moves `cmd_run` to `cli/run.py`, extract a `prepare_run()` function:

```python
def prepare_run(
    prd_id: str,
    data_dir: Path,
    workflows_dir: Path | None,
    repo_root: Path,
    *,
    base_ref: str | None = None,
    workflow_override: str | None = None,
    model_override: str | None = None,
) -> tuple[PRD, Workflow, str]:
    """Resolve all inputs needed to call run_workflow().

    Returns (prd, workflow, resolved_base_ref).
    """
```

Note: `data_dir` is `.darkfactory/data/` (the new PRD-622 layout); internally `load_one(data_dir, prd_id)` derives `data_dir/"prds"`.

`cmd_run` becomes a thin shell: parse args -> call `prepare_run()` -> call `run_workflow()` / `execute_graph()` -> format output.

Depends on PRD-556.16 because `cmd_run` must be in its own module first.

## Acceptance criteria

- [ ] AC-1: `prepare_run()` is callable without argparse context
- [ ] AC-2: `cmd_run` delegates to `prepare_run()` for argument resolution
- [ ] AC-3: The function is independently testable with plain arguments
- [ ] AC-4: Existing `tests/test_cli_run.py` tests pass
