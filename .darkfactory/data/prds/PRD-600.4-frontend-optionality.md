# Frontend optionality seams

## Summary

Low-cost architectural seams that preserve the option of adding a frontend/API layer without premature abstraction. Extracts argument resolution and reconcile domain logic from CLI handlers into reusable functions, completes `--json` coverage for read commands, and introduces a structured error hierarchy to replace `SystemExit` for non-CLI consumers.

## Motivation

The architectural review found that the domain layer (prd.py, graph.py, runner.py, workflow.py) is already well-separated and server-callable. The gap is in the CLI handlers where argument resolution, subprocess orchestration, and presentation are mixed. These seams benefit the CLI today (testability, crash recovery) while enabling a future API layer.

**Explicitly deferred** (per adversarial review): service layer for all 15 commands, JSON response envelopes, `to_dict()` on all dataclasses, typed args protocol.

Depends on PRD-556 (CLI modularization) because extracting logic from `cmd_run` and `cmd_reconcile` is easier once they're in their own modules.

## Acceptance criteria

- [ ] `cmd_run` argument resolution callable without argparse context
- [ ] Reconcile git/gh operations callable without CLI handler
- [ ] All read commands support `--json`
- [ ] `HarnessError` hierarchy replaces `SystemExit` in domain paths

## Assessment (2026-04-11)

- **Value**: 1/5 today — the PRD itself says these are "seams that
  preserve the option of adding a frontend/API layer without premature
  abstraction." There is no frontend consumer. The seams benefit the
  CLI marginally (testability), but that marginal benefit isn't worth
  the refactor cost against today's baseline.
- **Effort**: l as scoped — four children across argument resolution,
  reconcile domain extraction, broad `--json` coverage, and a new
  exception hierarchy.
- **Current state**: greenfield. `HarnessError` does not exist. No
  extracted argument-resolution function. No service layer between
  CLI handlers and domain.
- **Gaps to fully implement**: all of it. Each child PRD has its own
  assessment.
- **Recommendation**: defer the sub-epic as a whole. The PRD's own
  dependency on PRD-556 is effectively satisfied (modularization is
  done), but the motivation isn't — no frontend is on the near horizon.
  One partial exception: PRD-600.4.4 (`HarnessError` hierarchy) is
  worth landing opportunistically whenever a command gains `--json`
  output and needs structured error objects, because at that point
  you're already designing the error payload. Treat 600.4.4 as an
  opportunistic pairing, not a standalone task.
