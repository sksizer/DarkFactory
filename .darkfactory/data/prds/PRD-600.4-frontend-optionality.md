# Frontend optionality seams

## Summary

Low-cost architectural seams that preserve the option of adding a frontend/API layer without premature abstraction. Extracts argument resolution and reconcile domain logic from CLI handlers into reusable functions, completes `--json` coverage for read commands, and introduces a structured error hierarchy to replace `SystemExit` for non-CLI consumers.

## Motivation

The architectural review found that the domain layer (`model/`, `graph.py`, `runner.py`, `workflow.py`) is already well-separated and server-callable. The gap is in the CLI handlers where argument resolution, subprocess orchestration, and presentation are mixed. These seams benefit the CLI today (testability, crash recovery) while enabling a future API layer.

**Explicitly deferred** (per adversarial review): service layer for all 15 commands, JSON response envelopes, `to_dict()` on all dataclasses, typed args protocol.

Depends on PRD-556 (CLI modularization) because extracting logic from `cmd_run` and `cmd_reconcile` is easier once they're in their own modules.

## Acceptance criteria

- [ ] `cmd_run` argument resolution callable without argparse context
- [ ] Reconcile git/gh operations callable without CLI handler
- [ ] All read commands support `--json`
- [ ] `HarnessError` hierarchy replaces `SystemExit` in domain paths
