# Worktree lifecycle resilience: auto-recovery and pre-run cleanup

## Summary

Stale worktree/branch state is the single largest cause of workflow failure, responsible for 45% of PRD-level failures in analyzed runs. When a PR is merged or closed, the local branch persists but the worktree directory is gone. `ensure_worktree` detects this and raises, requiring manual `prd cleanup` — which is impossible during unattended batch execution.

This feature extracts a shared branch-cleanup API, then wires it into both `ensure_worktree` (reactive recovery) and the graph execution loop (proactive cleanup).

## Decomposition

1. **567.1.1** — Extract a `delete_branch_if_safe` helper into `builtins/_shared.py`. This is the shared contract: given a branch name and repo root, check PR state and delete the branch if the PR is merged/closed. No dependencies.

2. **567.1.2** — Wire auto-recovery into `ensure_worktree`. When the `branch exists + worktree missing` condition is detected, call the helper instead of raising. Depends on 567.1.1.

3. **567.1.3** — Add pre-run stale cleanup to graph execution. Before the DAG loop begins, iterate stale worktrees and clean them using the helper. Depends on 567.1.1 but NOT on 567.1.2 — both are independent consumers of the same API.

## Acceptance criteria

- [ ] AC-1: A shared helper exists for safe branch deletion with PR state checks.
- [ ] AC-2: `ensure_worktree` auto-recovers when the PR is merged/closed instead of raising.
- [ ] AC-3: `prd run --all --execute` cleans stale worktrees before starting the DAG loop.
