# Rework loop detection

## Summary

A safety check that detects when `prd rework PRD-X` produces no changes. If the agent runs but stages nothing, this indicates a potential loop: the agent thinks comments are already addressed, disagrees with the reviewer, or is confused. The harness must flag this loudly rather than silently no-op.

## Current State

No implementation exists. Blocked on PRD-225.4 (rework workflow).

## Note on Architecture

The original design proposed a standalone `rework_guard.py` at the package root with its own state file. This should instead be integrated into the runner or the rework workflow itself — the "did we produce changes?" check is a natural part of the commit step, not a separate module. State tracking (consecutive no-change counts) could live in the event log rather than a bespoke JSON file.

This also overlaps conceptually with PRD-609 (workflow retry/recovery) — loop detection is a special case of "should we retry or give up?"

## Requirements

1. After the agent runs in the rework workflow, check whether any files were modified (via `git diff --cached` or `git status`).
2. If no changes staged, emit a clear warning.
3. Track consecutive no-change rework attempts per PRD.
4. After N consecutive no-change attempts (default 2), escalate to error and block further automatic rework until manual intervention.
5. Reset counter when a rework produces changes.
6. Integrate with rework-watch daemon (PRD-225.6) so the watcher respects guard state.

## Acceptance Criteria

- [ ] Rework run with no changes emits a clear warning
- [ ] After N consecutive no-change runs, harness blocks further automatic rework
- [ ] Producing changes resets the counter
- [ ] Guard state persists across invocations
- [ ] Rework-watch daemon respects guard and skips blocked PRDs
- [ ] Tests cover: changes reset, warning escalation, blocking, state persistence, independent counters per PRD
