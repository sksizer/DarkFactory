# Align planning workflow with WorkflowTemplate

## Summary

The planning workflow (`planning/workflow.py`) builds its full task list manually with an inline `Workflow(tasks=[...])`, while the default and extraction workflows use `WorkflowTemplate.compose()`. This means the planning workflow's open/close sequences can drift from the SDLC invariants that templates enforce (e.g. `commit_events` ordering, `lint_attribution` placement, `summarize_agent_run` inclusion).

This task refactors the planning workflow to use a template — either the existing `PRD_IMPLEMENTATION_TEMPLATE` or a new `PLANNING_TEMPLATE` if the close sequence requires different ordering.

## Requirements

1. Determine if `PRD_IMPLEMENTATION_TEMPLATE` can be used directly (does the planning close sequence match?).
2. If not, create a `PLANNING_TEMPLATE` in `templates_builtin.py` with the planning-specific close sequence.
3. Refactor `planning/workflow.py` to use `template.compose(...)` instead of `Workflow(tasks=[...])`.
4. The resulting workflow must produce the same task list as before (same builtins in the same order), just constructed via the template mechanism.
5. Verify that `prd plan <epic>` dry-run output is unchanged.

## Technical approach

Compare the current planning workflow's task list with `PRD_IMPLEMENTATION_TEMPLATE`:

**Current planning open:**
```python
BuiltIn("ensure_worktree"),
BuiltIn("set_status", kwargs={"to": "in-progress"}),
BuiltIn("commit", kwargs={"message": "chore(prd): {prd_id} start decomposition"}),
```

**Template open:**
```python
BuiltIn("ensure_worktree"),
BuiltIn("set_status", kwargs={"to": "in-progress"}),
BuiltIn("commit", kwargs={"message": "chore(prd): {prd_id} start work"}),
```

The only difference is the commit message ("start decomposition" vs "start work"). This can be handled by either:
- Parameterizing the commit message in the template, or
- Accepting the template's default message (the message is cosmetic, not semantic).

**Current planning close:**
```python
BuiltIn("set_status", kwargs={"to": "review"}),
BuiltIn("commit_transcript"),
BuiltIn("commit", kwargs={"message": "chore(prd): {prd_id} decomposed into tasks"}),
BuiltIn("lint_attribution"),
BuiltIn("push_branch"),
BuiltIn("create_pr"),
```

**Template close:**
```python
BuiltIn("set_status", kwargs={"to": "review"}),
BuiltIn("commit_events"),
BuiltIn("commit", kwargs={"message": "chore(prd): {prd_id} ready for review"}),
BuiltIn("summarize_agent_run"),
BuiltIn("lint_attribution"),
BuiltIn("push_branch"),
BuiltIn("create_pr"),
```

Key differences:
- Planning has `commit_transcript`, template has `commit_events` — these are different builtins. Planning may need both.
- Planning lacks `summarize_agent_run` — should be added.
- Commit messages differ (cosmetic).

Decision: create a `PLANNING_TEMPLATE` that mirrors the standard template but accommodates the planning-specific close sequence (both `commit_transcript` and `commit_events`). Or, align the planning workflow to the standard template's close sequence if the differences aren't semantically important.

## Acceptance criteria

- [ ] AC-1: `planning/workflow.py` uses a `WorkflowTemplate.compose()` call instead of `Workflow(tasks=[...])`.
- [ ] AC-2: The planning workflow's open and close sequences are generated by the template, not manually listed.
- [ ] AC-3: `prd plan <epic>` dry-run output shows the same task sequence as before (modulo commit message wording).
- [ ] AC-4: The planning workflow includes `commit_events` and `summarize_agent_run` in its close sequence.
- [ ] AC-5: `just test`, `just lint`, `just typecheck` all pass.
