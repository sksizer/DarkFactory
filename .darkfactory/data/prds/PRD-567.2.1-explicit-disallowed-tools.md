# Add explicit --disallowed-tools for harness-owned git operations

## Summary

Currently `git commit`, `git push`, and `gh pr` are not in the `--allowed-tools` list, so Claude Code falls back to its interactive approval system when agents attempt them. In `--print` mode this produces a "requires approval" error that the agent then tries to work around, wasting 3-7 turns per session. Adding these to `--disallowed-tools` makes the rejection instant — the tool call is refused before being sent to the subprocess.

## Requirements

1. Add `--disallowed-tools` entries to `invoke.py` for operations the harness owns:
   - `Bash(git commit:*)`
   - `Bash(git push:*)`
   - `Bash(gh pr:*)`
   - `Bash(git checkout:*)`
   - `Bash(git switch:*)`
   - `Bash(git branch:*)`
2. These must be appended after the existing containment `--disallowed-tools` entries.
3. Verify that the deny list does not interfere with allowed tools (e.g. `Bash(git add:*)` should still work).
4. Unit test: verify the constructed command includes the new `--disallowed-tools` entries.

## Technical approach

```python
# In invoke.py, after line 462 (the existing --disallowed-tools for Read(../)):
# Deny harness-owned operations so denials are instant, not interactive.
"--disallowed-tools", "Bash(git commit:*)",
"--disallowed-tools", "Bash(git push:*)",
"--disallowed-tools", "Bash(gh pr:*)",
"--disallowed-tools", "Bash(git checkout:*)",
"--disallowed-tools", "Bash(git switch:*)",
"--disallowed-tools", "Bash(git branch:*)",
```
