# Graceful timeout handling for late result events

## Summary

When the timeout fires on an agent task, the harness currently marks the run as failed unconditionally. However, the agent process may have already emitted a terminal `result` event (as happened with PRD-541, which completed in ~770s but was killed at 600s). This task modifies the timeout handler in `invoke.py` to check for a terminal result event before declaring a timeout failure, preserving completed work that arrived just before the deadline.

## Requirements

1. In the timeout handler in `invoke.py`, before marking the run as failed, scan the captured stdout for a terminal `{"type":"result", ...}` JSON line.
2. If a terminal result event is found:
   - Use its `is_error` / `subtype` fields to determine success or failure, instead of the timeout verdict.
   - Log a warning: `"Task timed out at {timeout}s but a terminal result event was found — using result verdict instead."`
3. If no terminal result event is found, behave as today: mark the run as a timeout failure.
4. Unit tests covering both cases: timeout with result event present, and timeout with no result event.

## Technical Approach

### `python/darkfactory/invoke.py`

In the timeout handling path (around line 552 where the timeout failure is constructed), add a check:

```python
import json

def _find_terminal_result(output_lines: list[str]) -> dict | None:
    """Scan output lines for the last {"type": "result", ...} JSON object."""
    for line in reversed(output_lines):
        line = line.strip()
        if not line.startswith("{"):
            continue
        try:
            obj = json.loads(line)
            if obj.get("type") == "result":
                return obj
        except json.JSONDecodeError:
            continue
    return None
```

In the timeout branch:

```python
terminal_result = _find_terminal_result(captured_lines)
if terminal_result is not None:
    logger.warning(
        "Task timed out at %ds but terminal result event found — using result verdict",
        timeout_seconds,
    )
    # Build InvokeResult from terminal_result instead of timeout failure
    ...
else:
    # Existing timeout failure path
    ...
```

### Tests (`tests/test_invoke.py`)

- Test `_find_terminal_result` with: no result line, result line present, multiple JSON lines (picks the result one), malformed JSON lines.
- Integration-style test: simulate a timeout where captured output contains a result event; verify the run is reported based on the result event, not as a timeout.

## Acceptance Criteria

- [ ] AC-1: When a task times out but has emitted a terminal `result` event, the run outcome is based on that event (not a timeout failure).
- [ ] AC-2: When a task times out with no terminal result event, the run is reported as a timeout failure (existing behavior preserved).
- [ ] AC-3: A warning is logged when a terminal result event is found after timeout.
- [ ] AC-4: Unit tests cover both paths (result present, result absent) and edge cases (malformed JSON, multiple JSON lines).
